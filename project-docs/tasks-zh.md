# 执行待办事项清单

## 1. 执行策略

**实现方法**:
- 垂直切片构建：在进入下一个功能之前端到端完成一个功能
- 从核心基础设施开始（Electron + React + 存储）以实现快速迭代
- 先实现聊天模式（较简单），然后实现沉浸模式（更复杂）
- 增量集成AI服务（先文本，后图像）
- 保持小而可逆的提交，便于调试

**依赖逻辑**:
- 存储层必须在任何数据依赖功能之前存在
- 房间管理支持角色管理
- 角色管理支持消息传递
- 聊天模式必须在构建沉浸模式之前工作
- 文本生成必须在图像生成之前工作

**阶段划分**:
1. **基础**: 基础设施和基本UI
2. **核心功能**: 房间、角色、无AI的消息传递
3. **AI集成**: 文本和图像生成
4. **呈现模式**: 聊天和沉浸视图
5. **润色**: 错误处理、性能、UX优化

**早期验证**:
- 使用虚拟数据验证存储层
- 在AI集成之前验证UI布局
- 使用单个房间/角色测试AI集成
- 首先使用静态内容验证模式切换

## 6. AI编码推荐工作模式

在AI编码工具（Cursor、Copilot等）中使用此待办事项时：

1. **一次一个任务**: 每个会话实现恰好一个任务，不要合并
2. **首先阅读情境**: 开始前审查相关的构建规格部分
3. **小提交**: 遵循建议的提交粒度，每个逻辑步骤后提交
4. **立即验证**: 在移动到下一个任务之前测试每个任务完成
5. **无范围蔓延**: 仅实现任务范围内的内容，抵制添加"锦上添花"
6. **审查代码**: 检查生成的代码的类型安全、错误处理、一致性
7. **更新文档**: 如果实现与规格不同，更新build-spec.md
8. **跟踪进度**: 仅在满足所有验收标准时标记任务完成
9. **提出问题**: 如果任务不明确，在实现前澄清（在注释中询问）
10. **迭代**: 如果方法不起作用，回退并尝试替代方案（小提交有帮助）

**避免**:
- 将多个任务合并为一个实现
- 跳过错误处理"暂时"
- 提交大的、未测试的更改
- 实现不在待办事项中的功能
- 忽略验收标准

**最佳实践**:
- 严格使用TypeScript（启用 `strict` 模式）
- 优先使用现有模式而不是发明新模式
- 编写描述性的提交消息
- 保持组件小而专注
- 早期且经常用真实数据测试
