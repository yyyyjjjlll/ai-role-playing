# 产品简报

## 1. 产品概述
- **产品定位**: 一个基于AI的角色扮演与剧情生成平台，使用户能够创建自定义角色、构建世界观，并参与由AI驱动的动态故事情节
- **目标用户**: 角色扮演爱好者、创意作家、故事讲述者和互动小说爱好者
- **核心问题**: 传统的角色扮演体验需要大量的手动叙事工作，缺乏具有灵活参与模式的动态AI驱动叙事生成
- **为何现在**: 先进的AI语言模型现已实现实时、情境化的叙事生成，使沉浸式AI驱动的角色扮演体验变得可行且引人入胜

## 2. 目标
- **主要目标**: 使用户能够以最小的设置摩擦创建和参与AI驱动的角色扮演体验
- **次要目标**:
  - 支持主动参与（演绎者）和被动观察（旁观者）两种模式
  - 为不同用户偏好提供双重呈现模式（聊天和沉浸式）
  - 启用AI辅助的角色和视觉资产生成
- **成功指标**:
  - 用户在5分钟内成功创建并加入聊天室
  - 70%以上的用户尝试两种呈现模式
  - AI生成内容（角色、背景）接受率 > 60%

## 3. 目标用户
- **主要用户**:
  - 18-35岁喜欢角色扮演游戏的创意人群
  - 寻求协作式故事创作工具的作家
  - 互动小说爱好者
- **关键用户特征**:
  - 熟悉聊天界面
  - 对AI辅助创意内容感兴趣
  - 愿意尝试不同的叙事模式
- **核心使用场景**:
  - 与朋友创建自定义RPG场景
  - 与AI同伴进行单人创意写作
  - 观察和引导AI生成的叙事

## 4. 问题陈述
- **当前痛点**:
  - 传统RPG平台需要大量的手动故事创作工作
  - 参与模式灵活性有限（必须主动扮演）
  - 角色和视觉资产创建耗时
  - 缺乏动态、情境感知的叙事生成
- **现有替代方案**:
  - Character.AI（有限的世界构建，无多角色场景）
  - 传统桌面RPG平台（手动操作，非AI驱动）
  - 文本RPG游戏（固定叙事，有限定制）
- **为何不足**:
  - 没有多角色管理、双重参与模式和双重呈现风格的组合
  - 有限的AI驱动视觉生成（背景、角色立绘）
  - 缺乏灵活的用户身份切换

## 5. MVP范围

### 包含范围
- **聊天室管理**:
  - 左侧面板聊天室列表
  - 创建/加入聊天室
  - 房间特定设置

- **角色创建**:
  - 每个房间可创建多个角色
  - 角色名称、描述/设定
  - 角色立绘上传（图片文件或URL）
  - AI生成角色立绘（如果用户未提供）

- **用户身份选择**:
  - 演绎者（Actor）: 拥有指定角色的主动参与者
  - 旁观者（Observer）: 具有引导能力的被动观察者

- **双重呈现模式**:
  - **聊天模式**: 支持旁白的文本聊天界面
  - **沉浸模式**: 视觉小说风格，带角色立绘和AI生成背景

- **AI驱动功能**:
  - AI生成的剧情推进
  - AI生成的角色立绘（可选）
  - 基于故事情境的AI生成场景背景

### 不包含范围
- 多用户实时协作（先专注于单用户或异步场景）
- 语音旁白或音频功能
- 复杂的战斗/掷骰子机制
- 角色属性系统
- 导出到其他格式（PDF、视频等）
- 移动应用（Web优先）
- 货币化功能
- 用户认证（可在MVP后添加）

## 6. 用户流程

1. **入口**: 用户打开应用程序
2. **房间选择**: 用户在左侧面板看到聊天室列表
3. **房间创建/加入**:
   - **创建**: 点击"新建房间" → 输入房间名称和设置
   - **加入**: 从列表中点击现有房间
4. **角色设置**:
   - 添加角色的名称和描述
   - 上传立绘图片 或 留空由AI生成
   - 将角色分配给自己（如果是演绎者）或保持未分配
5. **身份选择**: 选择身份为演绎者或旁观者
6. **模式选择**: 在聊天模式和沉浸模式之间切换
7. **互动**:
   - **聊天模式**: 发送消息，AI用角色对话和旁白回应
   - **沉浸模式**: 以视觉小说格式查看角色立绘、背景和对话
8. **剧情推进**: AI生成情境响应，在沉浸模式中更新背景
9. **完成**: 用户可以随时保存/退出房间

## 7. 功能需求

1. **聊天室系统**:
   - 在左侧边栏显示所有用户房间
   - 支持自定义命名的房间创建
   - 持久化房间状态

2. **角色管理**:
   - 每个房间创建多个角色
   - 必填字段：名称、描述
   - 可选：立绘图片（上传或URL）
   - 缺失立绘的AI回退

3. **身份系统**:
   - 在演绎者和旁观者之间切换
   - 演绎者：分配给特定角色
   - 旁观者：可以发送引导消息

4. **聊天模式呈现**:
   - 基于文本的消息列表
   - 支持旁白消息（系统样式）
   - 带角色名称标签的角色消息

5. **沉浸模式呈现**:
   - 显示当前角色立绘
   - 显示AI生成的背景图像
   - 以视觉小说风格呈现对话
   - 场景之间的平滑过渡

6. **AI集成**:
   - 基于情境生成叙事响应
   - 未提供时生成角色立绘
   - 动态生成场景背景
   - 维护对话情境

## 8. 非功能性期望

- **性能**:
  - AI叙事生成响应时间 < 5秒
  - 图像生成 < 15秒
  - 平滑的模式切换（< 1秒）

- **可用性**:
  - 直观的类聊天界面
  - 模式之间的清晰视觉区分
  - 简单的角色创建过程

- **响应性**:
  - Web响应式设计（桌面优先）
  - 在较小屏幕上优雅降级

- **可靠性**:
  - 持久化聊天历史和房间状态
  - 优雅处理AI服务故障

- **安全/隐私**:
  - 清理用户输入
  - 安全的图像上传处理
  - MVP的本地存储（隐私优先）

## 9. 风险和开放问题

**产品风险**:
- 用户期望不匹配（AI质量 vs 用户期望）
- 双模式UI的复杂性可能会使用户困惑
- 角色创建摩擦可能降低参与度

**技术风险**:
- AI生成延迟影响用户体验
- 图像生成成本和速度权衡
- 长叙事的情境窗口限制
- 背景生成质量一致性

**UX风险**:
- 模式切换可能造成干扰
- 沉浸模式可能感觉与聊天历史脱节
- 观察者角色可能感觉太被动

**开放问题**:
- 房间是否应支持多个同时在线用户？（MVP范围外）
- 如何处理非常长的对话历史？
- 对AI生成的用户控制级别是多少？
- 是否应缓存/重用背景？

## 10. 验收标准

- 用户可以创建具有自定义名称的新聊天室
- 用户可以向房间添加至少2个带有名称和描述的角色
- 用户可以上传角色立绘 或 查看AI生成的立绘
- 用户可以选择身份为演绎者或旁观者
- 用户可以在聊天模式和沉浸模式之间切换
- 在聊天模式中：消息显示角色名称和旁白文本
- 在沉浸模式中：角色立绘和背景正确显示
- AI在5秒内生成情境叙事响应
- AI生成与场景情境匹配的背景
- 房间状态在会话之间持久保存
- 左侧面板显示所有创建的房间
- 用户可以在房间之间切换而不丢失数据
